# Plan 1: Wire assembleSystemPrompt into OrchestratorService

## Objective

Replace the static inline system prompt string in `OrchestratorService.runScenario()` with a call to `assembleSystemPrompt(mcpId, tools)` from `src/shared/llm/systemPrompt.ts`, so each MCP gets a dynamic system prompt containing its actual tool names from the registry.

## Context

**Dead module:** `src/shared/llm/systemPrompt.ts` exports `assembleSystemPrompt(mcpId, allTools, basePromptTemplate?)` which:
- Looks up the MCP in the registry via `getRegistryEntry(mcpId)`
- Filters tools by `toolNamespacePrefix`
- Injects tool list into a prompt template

**Current call site:** `OrchestratorService.runScenario()` at lines 43-48 builds a hardcoded system message:
```typescript
content: `You are a test automation agent. Execute the following Gherkin scenario steps and report results.
Scenario: ${scenario.name}`,
```

**What needs to change:** Replace that static string with `assembleSystemPrompt(ctx.mcpConfig.id, [])` (empty tools array initially; the real tool list comes from MCP capability negotiation which is Phase 8). The key wiring is that the function IS called and the mcpId IS passed, producing an MCP-specific prompt.

## Implementation Steps

1. **Add import** at top of `src/server/orchestrator/OrchestratorService.ts`:
   ```typescript
   import { assembleSystemPrompt } from "../../shared/llm/systemPrompt";
   ```

2. **Replace the static system message** (lines 43-48) with:
   ```typescript
   const systemPromptContent = assembleSystemPrompt(ctx.mcpConfig.id, []);
   const initialMessages: LLMMessage[] = [
     {
       role: "system",
       content: `${systemPromptContent}\nScenario: ${scenario.name}`,
     },
     ...ctx.conversationHistory,
   ];
   ```
   Note: Pass `[]` as tools for now. When Phase 8 wires real MCP capability negotiation, the tool list will be populated. The important thing is that `assembleSystemPrompt` is called with the correct `mcpId`, producing a per-MCP prompt (even if tools list is empty, the prompt template differs from the old hardcoded one).

3. **Update tests** in `src/server/orchestrator/OrchestratorService.test.ts`:
   - The existing tests use `MockLLMProvider` which receives the `request.messages` array. Verify that tests still pass (the system message content changed but tests don't assert on system message content).
   - Add one new test that verifies the system message is NOT the old hardcoded string:
   ```typescript
   test("runScenario uses assembleSystemPrompt instead of static prompt", async () => {
     let capturedMessages: LLMMessage[] = [];
     const capturingProvider: LLMProvider = {
       async complete(request: LLMRequest): Promise<LLMResponse> {
         capturedMessages = request.messages;
         return { /* standard mock response */ };
       },
       async *stream() { yield { index: 0, delta: "", finishReason: "stop" }; },
       async estimateCost() { return 0; },
     };
     const orchestrator = new OrchestratorService(capturingProvider);
     // ... run with a 1-step scenario ...
     const systemMsg = capturedMessages.find(m => m.role === "system");
     assert.ok(systemMsg);
     assert.ok(!systemMsg.content.includes("You are a test automation agent"));
     assert.ok(systemMsg.content.includes("browser automation assistant") || systemMsg.content.includes("tools"));
   });
   ```
