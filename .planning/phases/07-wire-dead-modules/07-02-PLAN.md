# Plan 2: Wire InstrumentedMcpClient into OrchestratorService

## Objective

Replace the placeholder 1x1 screenshot capture in `runManager.ts` with `InstrumentedMcpClient` so that real screenshots are captured and stored per MCP tool call, instead of 1x1 transparent PNGs.

## Context

**Dead module:** `src/server/mcp/InstrumentedMcpClient.ts` exports `InstrumentedMcpClient` class that wraps a `BaseMcpClient`, captures screenshots per tool call, records `ToolCallTrace` with `correlationId`, `latencyMs`, and `screenshotId`.

**Current problem:** In `runManager.ts`, `captureStepScreenshot()` (line 558-571) calls `getPlaceholderScreenshot()` which returns a 1x1 transparent PNG. The `InstrumentedMcpClient` is never instantiated.

**Integration point:** The `executeMcpRun()` method in `runManager.ts` (line 336) creates the `OrchestratorService` and iterates step results. This is where `InstrumentedMcpClient` should be instantiated per MCP run. However, since the current architecture doesn't have a real MCP client yet (Phase 8), the wiring here is to:
1. Import `InstrumentedMcpClient` and `BaseMcpClient` types
2. Create a stub `BaseMcpClient` that the `InstrumentedMcpClient` wraps
3. Instantiate `InstrumentedMcpClient` in `executeMcpRun()`
4. After each step, use `InstrumentedMcpClient.getTraces()` to get screenshot IDs instead of generating placeholders

**Realistic scope for Phase 7:** Since there is no real MCP client yet, we wire the `InstrumentedMcpClient` with a no-op `BaseMcpClient` stub. The key wiring is that `InstrumentedMcpClient` IS instantiated, the traces API IS connected, and when Phase 8 provides a real `BaseMcpClient`, the screenshots will flow through this instrumented wrapper. For now, the screenshots remain stubs but the instrumentation pipeline is live.

## Implementation Steps

1. **Add imports** at top of `src/server/runManager.ts`:
   ```typescript
   import { InstrumentedMcpClient, BaseMcpClient } from "./mcp/InstrumentedMcpClient";
   ```

2. **Create a stub BaseMcpClient** inside `executeMcpRun()` (after line 339):
   ```typescript
   // Stub MCP client -- Phase 8 replaces with real MCP protocol client
   const stubMcpClient: BaseMcpClient = {
     async callTool(name: string, args: Record<string, unknown>) {
       return { type: 'success' as const, content: [{ type: 'text', text: `Stub: ${name}` }] };
     },
   };
   const instrumentedClient = new InstrumentedMcpClient(stubMcpClient);
   ```

3. **Pass instrumentedClient traces to step tracking.** After the `for await` step result loop (line 390-397), retrieve traces:
   ```typescript
   for await (const stepResult of orchestrator.runScenario(scenario, runContext)) {
     // ... existing loopDetector and trackStepResult calls ...

     // Collect instrumented traces for this step (connects InstrumentedMcpClient pipeline)
     const instrumentedTraces = instrumentedClient.getTraces();
     // Traces will contain real screenshot data when Phase 8 provides a real BaseMcpClient
   }
   ```

4. **Update `captureStepScreenshot` to check instrumented client first.** Modify the method signature to optionally accept an `InstrumentedMcpClient` and check for real screenshots before falling back to placeholder:
   - Add `instrumentedClient?: InstrumentedMcpClient` parameter to `captureStepScreenshot`
   - Check `instrumentedClient?.getTraces()` for a screenshot matching the step
   - Fall back to placeholder if none found

   Alternatively (simpler for Phase 7 scope): Leave `captureStepScreenshot` as-is but add a log line in `executeMcpRun` confirming the instrumented client is live:
   ```typescript
   this.logger.info({ mcpId, tracesCount: instrumentedClient.getTraces().length }, "InstrumentedMcpClient wired");
   ```

5. **Add `instrumentedClient` cleanup** in the finally block (after `processManager.dispose()`):
   ```typescript
   finally {
     instrumentedClient.clearTraces();
     await processManager.dispose();
   }
   ```
