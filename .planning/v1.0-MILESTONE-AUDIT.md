## Cross-Phase Integration Findings

### Root Cause: Three Dead-Code Modules in Phase 3

The most impactful gap is that three Phase 3 modules were implemented and tested in isolation but were never wired into the execution path:

1. **`InstrumentedMcpClient`** (`src/server/mcp/InstrumentedMcpClient.ts`) — the independent screenshot-capture proxy
2. **`assertionsRunner`** (`src/server/validation/assertionsRunner.ts`) — the independent Then-step validator
3. **`stalenessRecovery`** (`src/server/mcp/stalenessRecovery.ts`) — the stale ARIA ref retry handler

These three modules, plus `assembleSystemPrompt` from Phase 2, constitute the core anti-hallucination differentiation of this platform. Their absence means:
- All screenshots are 1×1 placeholders
- Then-step assertions never run
- ARIA staleness always causes step failure
- All MCPs receive the same generic system prompt

### Root Cause: Vision Validator is Heuristic-Only

`validateStepWithVision()` in `src/server/validation/visionValidator.ts` is a synchronous deterministic function with no image buffer parameter and no LLM API call. VALID-03, VALID-06, VALID-07 all describe an async LLM vision call that does not exist in the code.

### Root Cause: MCP Process is a Stub

`McpProcessManager.spawn()` starts `node -e 'setInterval(() => {}, 1 << 30)'` — a keep-alive process with no MCP protocol. The orchestrator never connects to this process; it calls the LLM directly and treats non-error LLM responses as step-passed results.
